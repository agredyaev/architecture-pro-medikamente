# Метрики качества и масштабирования

## 1. Метрики качества классификации

| Метрика | Назначение | Целевое значение |
| --- | --- | --- |
| `Precision` по классам `pii.basic`, `pii.extended`, `medical.special`, `financial` | Снизить долю ложноположительных классификаций | `>=0.95` |
| `Recall` по `medical.special` и `pii.extended` | Не допустить пропуск чувствительных полей | `>=0.98` |
| `F1-score` по каждому классу | Баланс precision и recall | `>=0.95` |
| `False Negative Rate` для `medical.special` | Контроль самого опасного сценария | `<=0.02` |
| Доля автоматически классифицированных наборов | Оценка автоматизации | `>=0.85` |
| Доля наборов в `Quarantine` | Оценка `schema drift` и неразмеченных источников | `<=0.10` в стабильном режиме |
| Доля публикаций, прошедших `Privacy policy gate` | Исключить обход consent/legal basis/stop processing | `100%` |
| Полнота `lineage` | Полнота источника, преобразования и получателя | `100%` |
| Время обнаружения `schema drift` | Скорость реакции на изменение схемы | `<=5m` |

## 2. Метрики масштабирования

Базовый нагрузочный профиль:

1. `1x` — `20 GB/day`, до `2M` records/day, `5` источников.
2. `3x` — `60 GB/day`, до `6M` records/day, `12` источников.
3. `5x` — `100 GB/day`, до `10M` records/day, `20` источников.

| Метрика | Назначение | Целевое значение |
| --- | --- | --- |
| Пропускная способность ingestion | Сколько наборов или ГБ обрабатывается в час | При переходе `1x -> 5x` пропускная способность масштабируется не хуже чем линейно по числу worker-ов в пределах целевого профиля |
| `p95` задержки пакетной классификации | Контроль задержки классификации | `<=15m` для batch `100 GB` или `10M` records |
| Очередь обработки | Контроль накопления backlog | `<=10m` при профиле `5x` |
| Максимум одновременных источников | Контроль расширения ландшафта | Не менее `20` одновременных источников при `<=10m` backlog |
| Время повторной обработки после сбоя | Устойчивость | `<=30m` |
| Рост хранения по слоям | Планирование емкости | Отдельный план capacity по `landing/restricted/curated/mart` |

## 3. Меры масштабирования

1. Горизонтальное масштабирование ingestion worker-ов.
2. Выделение `Schema profiler`, `Classifier`, `Pseudonymization` и `Export` в отдельные scale units.
3. Очереди между ingestion, classification и transform для сглаживания пиков.
4. Отдельный контур `Quarantine` для проблемных наборов без блокировки основного pipeline.
5. Разделение хранилищ по слоям и независимые политики retention.
6. Разделение плоскости метаданных и плоскости данных.
7. Переобучение модели и обновление правил без остановки основного ingestion.

## 4. Ограничения

1. При `confidence` ниже порога набор не должен публиковаться автоматически.
2. При `schema drift` без совместимости публикация в `Mart` запрещена.
3. При отсутствии `legal basis` или при `stop_processing=true` загрузка блокируется до удаления или ручного решения.
