## 2. Разделение методов защиты

| Метод | Где применять | Назначение |
| --- | --- | --- |
| Шифрование | Базы, документы, резервные копии, каналы передачи | Защита at rest и in transit |
| Маскирование | UI, рабочие места, отчеты ограниченного доступа | Скрытие части значений при сохранении полезности |
| Обфускация | Техлоги, сервисные трассы, временные файлы, support dumps | Исключение прямого чтения чувствительных значений в служебных артефактах |
| Псевдонимизация | Аналитика, интеграции с ограниченным набором идентификаторов | Замена прямых идентификаторов на устойчивые surrogate values |
| Обезличивание | Агрегированные отчеты и датасеты без необходимости обратного восстановления | Исключение идентификации субъекта |

## 3. Механизм тегирования

Тегирование должно назначаться на запись, документ, результат анализа, выгрузку и API-ответ.

### 3.1 Обязательные теги

| Тег | Назначение | Пример |
| --- | --- | --- |
| `tag.class` | Класс чувствительности | `pii.basic`, `medical.special`, `financial` |
| `tag.domain` | Домен-владелец данных | `patient`, `appointment`, `billing`, `laboratory`, `analytics` |
| `tag.subject_scope` | Объем субъектов | `single-patient`, `multi-patient`, `aggregate` |
| `tag.legal_basis` | Основание обработки | `contract`, `medical-care`, `consent` |
| `tag.retention` | Срок хранения | `1y`, `5y`, `medical-long-term` |
| `tag.masking` | Правило маскирования | `none`, `partial`, `full` |
| `tag.export` | Ограничение экспорта | `forbidden`, `controlled`, `allowed-aggregate` |
| `tag.audit` | Обязательность усиленного аудита | `standard`, `strict` |

### 3.2 Точки назначения тегов

1. При создании профиля пациента.
2. При создании записи на прием.
3. При загрузке или генерации документа.
4. При приеме результата лаборатории.
5. Перед публикацией API-ответа.
6. Перед загрузкой в аналитический контур.
7. Перед выгрузкой отчета пользователю.

### 3.3 Правила применения

1. `medical.special` всегда получает `tag.audit=strict`.
2. `medical.special` и `pii.extended` не должны попадать в аналитическую витрину без псевдонимизации или обезличивания.
3. Любой объект с `tag.export=forbidden` не может быть отдан внешнему API и не может быть выгружен в файл пользователем.
4. Для документов с `tag.retention` должна быть настроена автоматическая политика удаления или архивирования.
5. Документы с `tag.class=medical.special` и `tag.domain=patient` недоступны роли `RECEPTION`; ресепшен видит только договорные и административные документы по object policy.

## 4. Инструменты, способы и меры по потокам

| Процесс | Меры на входе | Меры в обработке | Меры в хранении | Меры при передаче | Меры контроля |
| --- | --- | --- | --- | --- | --- |
| `P1` Запись на прием | Минимизация полей, валидация ввода | RBAC/ABAC, канонический patient ID | Шифрование реестра пациента и записи | TLS | Audit trail, проверка несогласованных изменений |
| `P2` Медданные и документы | Разделение профиля и медкарты | Тегирование, object-level access по типу документа, masking | Раздельные хранилища, versioning, retention | TLS | Аудит чтения, журнал удаления, аномалии доступа |
| `P3` Оплата | Исключение локального Excel | Биллинг как единая точка записи | Шифрование реестра оплат | Контрактный адаптер к ККМ и 1С | Сверка событий, аудит транзакций |
| `P4` Лаборатория | Минимизация полей в запросе | Нормализация и валидация ответа | Отдельный реестр направлений и результатов | API, mTLS, контрактная схема | Audit trail, lineage результата |
| `P5` Аналитика | Отбор разрешенных наборов | Классификация, псевдонимизация, запрет прямого экспорта | Raw / restricted / curated зоны, шифрование | Контролируемый ingestion | Lineage, контроль выгрузок, аудит чтения, monitoring/anomaly, метрики качества |
| `P6` ТМЦ и закупки | Исключение patient и medical data | RBAC по роли и филиалу, тегирование internal/financial | Шифрование реестров и документов | Контрактный адаптер в 1С | Audit trail закупки, приемки и списания |

## 5. Перечень базовых инструментов и практик

1. Канонические реестры пациентов, записей, направлений, результатов и оплат.
2. RBAC для базовых ролей и ABAC для данных уровня пациента, врача, филиала и домена.
3. Шифрование at rest для баз данных, документов и резервных копий.
4. TLS или mTLS для всех внутренних и внешних API.
5. Сервис тегирования и policy enforcement перед чтением, экспортом и аналитической загрузкой.
6. Audit trail для чтения, изменения, удаления, экспорта и интеграционных вызовов.
7. Lifecycle-правила: retention, архивирование, удаление, фиксация основания удаления.
8. Псевдонимизация и обезличивание для аналитики и обмена вне медицинского контура.
9. API-контракты с явным исключением лишних полей и проверкой схемы.
10. Data Lineage для производных наборов и интеграционных потоков.
11. Consent registry и проверка legal basis перед чтением, экспортом и внешней передачей.
12. Stop processing по запросу субъекта и каскадное применение delete policy к связанным данным.
13. Controlled export для операционных и аналитических контуров.

## 6. Связь с принципами Privacy by Design

| Принцип | Что нужно сделать в Task1 |
| --- | --- |
| Privacy by Design | Встроить контроль доступа, аудит, минимизацию и lifecycle в сам поток данных |
| Data Minimization | Собирать только поля, нужные для конкретного процесса и роли |
| Data Lineage | Фиксировать происхождение данных, их преобразование и экспорт |
