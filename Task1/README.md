# Task1 — анализ безопасности данных

## 1. Цель

Зафиксировать As-Is потоки данных, выявить неучтенные конфиденциальные данные, описать проблемные зоны и меры защиты для To-Be.

## 2. Источники

- Исходная постановка проекта
- Схема текущего ландшафта компании

## 2.1 Нотация

Диаграммы выполнены в draw.io; каждая схема показывает акторов, системы, хранилища, операции над данными и меры контроля. Для `P5` To-Be представлен двумя схемами: `pipeline` и `governance`.

## 3. Охват

| Код | Процесс | Причина включения |
| --- | --- | --- |
| `P1` | Запись на прием | Обрабатываются идентификационные и контактные данные пациента |
| `P2` | Регистрация пациента и ведение меддокументов | Обрабатываются ПДн, расширенные ПДн, медданные, документы |
| `P3` | Прием оплаты и учет платежей | Обрабатываются платежные и договорные данные, есть дублирование учета |
| `P4` | Обмен с лабораторией и хранение результатов | Обрабатываются назначения, результаты анализов, медданные |
| `P5` | Аналитическая выгрузка и построение отчетности | Возникают копии чувствительных и производных данных вне операционного контура |
| `P6` | Учет ТМЦ и закупок | Обрабатываются внутренние, договорные и финансовые данные переходного legacy-контура |

`P6` относится к transition scope; отдельная wave procurement-domain не входит в текущий cutover.

## 4. Конфиденциальные данные, не учтенные во внутренних системах

Под «не учтены» понимаются данные без канонического хранилища или схемы, либо без классификации, владельца, истории доступа, срока хранения и цели обработки.

| Код | Набор данных | Где возникают в As-Is | Пробел учета |
| --- | --- | --- | --- |
| `U1` | Адрес регистрации, место работы или учебы | Расширенная анкета пациента | Нет отдельной структуры, нет классификации, нет цели хранения по полям |
| `U2` | Хронические заболевания | Расширенная анкета, файлы пациента | Нет домена медданных, нет разметки special category |
| `U3` | Сканы договоров, заключений, анализов | Папки пациента, JPG, PDF, scan | Нет схемы, нет версионирования, нет управления доступом по объекту |
| `U4` | Статус согласия на обработку, отзыв согласия, ограничение обработки | Бизнес-процесс есть, системного учета нет | Нельзя проверить законность дальнейшей обработки |
| `U5` | История доступа к медкартам и файлам | Любой процесс чтения файлов и Excel | Нет аудита действий и доказуемости доступа |
| `U6` | Теги конфиденциальности и правила маскирования | Любой поток | Нет механизма автоматического применения политик |
| `U7` | Срок хранения, дата удаления, факт удаления | Документы, Excel, выгрузки | Нет lifecycle-управления и контроля удаления |
| `U8` | Метаданные передачи в лабораторию | Назначение, отправка, получение результата | Нет трассировки заказа, получателя, версии результата |
| `U9` | Производные аналитические наборы | Jupyter, Excel-отчеты, локальные копии | Нет lineage, нет контроля экспорта, нет псевдонимизации |
| `U10` | Канонический идентификатор пациента | Все процессы | Нет единого мастер-ключа и надежного связывания записей |
| `U11` | Lawful basis и stop-processing state для аналитических публикаций | Ingestion-контур, restricted/curated/mart | Нельзя доказать правомерность аналитической публикации и вовремя остановить обработку |

## 5. Классы данных

| Тег | Класс | Состав |
| --- | --- | --- |
| `public` | Публичные данные | Справочники услуг, общие тексты без ПДн |
| `internal` | Внутренние служебные данные | Операционные статусы, служебные идентификаторы без ПДн |
| `pii.basic` | Базовые ПДн | ФИО, дата рождения, телефон, e-mail |
| `pii.extended` | Расширенные ПДн | Адрес, место работы или учебы, дополнительные анкеты |
| `medical.special` | Специальные категории ПДн | Жалобы, диагнозы, хронические заболевания, анализы, заключения |
| `financial` | Финансовые данные | Оплаты, чеки, счета, проводки, договорные атрибуты |
| `analytics.derived` | Производные аналитические данные | Витрины, выгрузки, feature-наборы, отчеты |

## 6. Проверяемые правила по процессам

| Процесс | Разрешенный минимальный набор данных | Явный запрет | Проверяемое условие |
| --- | --- | --- | --- |
| `P1` | ФИО, телефон, дата рождения, слот, специалист, филиал | Адрес, место работы, хронические заболевания, файлы пациента | Запись не создается, если payload содержит поля вне набора `P1-MIN-01` |
| `P2` | Профиль пациента, согласие, медкарта, документ по объектной ссылке | Доступ ресепшен к содержимому медкарты и медицинским документам, выгрузка документа без policy check | Объект `P2` доступен только при полном наборе тегов; роль `RECEPTION` не читает `medical.special`; каждое чтение объекта пишет `audit`-событие |
| `P3` | Patient ID, услуга, сумма, способ оплаты, статус, чек | Диагноз, жалобы, результаты анализов в биллинге | Биллинг не принимает поля класса `medical.special` |
| `P4` | Patient ID, код анализа, код назначения, филиал, версия результата | Сканы договора, расширенная анкета, лишние профильные поля | Лабораторный контракт отклоняет поля вне схемы `P4-LAB-01`; каждый результат имеет запись в lineage |
| `P5` | Только наборы с `tag.export!=forbidden`, valid `tag.legal_basis` и `stop_processing=false` | Загрузка субъектов с отзывом согласия, прямые идентификаторы в витрине, свободный экспорт raw/restricted | Публикация в `restricted` и `mart` разрешена только при `consent=pass`, `policy=pass`, `classifier=pass` |
| `P6` | Supplier ID, номенклатура, количество, цена, филиал, документ закупки | Patient ID, диагноз, результаты анализов | Контур закупок не принимает классы `pii.basic` и `medical.special` |
