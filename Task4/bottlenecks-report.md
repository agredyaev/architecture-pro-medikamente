# Отчет по узким местам миграции

## 1. Целевая проблема

Целевая проблема миграции: перенос ландшафта `Excel + файловый диск + 1С в файловом режиме + ручные обмены` в управляемую платформу без потери данных, без несанкционированного раскрытия `medical.special` и без остановки ключевых бизнес-процессов.

## 3. Вывод по причинам

1. Главный источник риска находится в данных, а не в коде: нет master data, source of truth и управляемого жизненного цикла.
2. Второй источник риска — эксплуатационный: нет формализованного cutover, rollback и подтвержденной восстановимости.
3. Третий источник риска — безопасность: privacy-механизмы должны мигрировать раньше аналитики и внешних интеграций.

## 4. Рекомендуемые меры

| Приоритет | Мера | Закрываемые категории | Ожидаемый эффект |
| --- | --- | --- | --- |
| `P1` | Выпустить master data model и канонический `patient ID` до бизнес-cutover | Данные и хранилища | Снижение ошибок сопоставления и rollback-риск |
| `P1` | Вывести Excel и файловый диск из роли первичного учета по доменам `P1-P5` | Данные и legacy | Снижение конфликтов версий и ручных расхождений |
| `P1` | Внедрить `IAM`, `RBAC/ABAC`, `Consent Registry`, `Policy Engine`, `Audit Trail`, `KMS` до switch of record | Безопасность | Закрытие критичных privacy-рисков |
| `P1` | Перевести лабораторию, ККМ и `1С` на контрактные адаптеры | Интеграции и legacy | Снижение потерь событий и лишних полей |
| `P1` | Формализовать доменный rollback и go/no-go criteria | Эксплуатация | Управляемый cutover без хаотичного отката |
| `P1` | Провести restore drill и репетицию reconciliation до запуска parallel run | Инфраструктура и эксплуатация | Подтверждение восстановимости и целостности |
| `P2` | Перевести аналитику на ingestion через restricted zone, classifier и pseudonymization | Интеграции и безопасность | Исключение локальных неконтролируемых выгрузок |
| `P2` | Включить DLP, controlled export, endpoint hardening, log redaction | Безопасность | Снижение риска утечки через локальные копии |
| `P2` | Формализовать runbook, роли и обучение пользователей по wave-переходам | Организация | Снижение операционных ошибок |
| `P2` | Включить мониторинг производительности, очередей, ошибок интеграций и аномалий доступа | Инфраструктура и эксплуатация | Быстрое обнаружение деградации |
| `P3` | Сократить остаточные legacy-write paths после стабилизации | Legacy | Упрощение сопровождения |
| `P3` | Выделить supplier-domain в отдельный procurement-контур только после стабилизации patient-data platform; до этого сохранить его в переходном `Legacy 1С` контуре | Организация и legacy | Снижение смешения областей миграции без расширения объема текущего cutover |

## 4.1 Правила присвоения приоритета

| Приоритет | Правило |
| --- | --- |
| `P1` | Риск блокирует cutover или затрагивает `medical.special`, source of truth, rollback, платежи, лабораторию, consent |
| `P2` | Риск не блокирует cutover, но обязателен к закрытию до завершения stabilization |
| `P3` | Риск не блокирует cutover `P1-P5`; мера выполняется после стабилизации основных доменов |

## 4.2 Exit criteria по мерам

| Мера | Exit criterion |
| --- | --- |
| Выпустить master data model и канонический `patient ID` | `100%` записей pilot-wave имеют канонический идентификатор и успешный reconciliation |
| Вывести Excel и файловый диск из роли первичного учета | Для доменов `P1-P5` отсутствуют write-операции в Excel и file share как source of truth |
| Внедрить privacy/access layer | `100%` критичных операций проходят через `IAM`, `policy`, `audit`, `consent`, `KMS` |
| Перевести лабораторию, ККМ и `1С` на контрактные адаптеры | `100%` интеграционных вызовов проходят contract validation и фиксируются в audit |
| Формализовать доменный rollback и go/no-go | Для каждой wave утверждены `Go Gate`, `Rollback trigger`, `rollback owner`, `max rollback time` |
| Провести restore drill и репетицию reconciliation | Restore и reconciliation завершены успешно в согласованные SLA |
| Перевести аналитику на restricted ingestion | `100%` наборов в аналитике проходят classifier, policy gate и lineage |
| Включить DLP и controlled export | `100%` чувствительных экспортов идут через controlled export; прямые выгрузки блокируются |
| Формализовать runbook и роли | По каждой wave есть утвержденный runbook и named owner |
| Включить мониторинг cutover | Доступны метрики lag, errors, throughput и audit completeness по всем волнам |
